# Audio - 设计与约束

> 对应 README：[README.md](./README.md)
> 文档最后更新：2026-02-27

## 设计目标与非目标

- **目标**：提供统一的音效与震动反馈管理，支持根据物理参数（力度、冲量）自动选择音效类型，支持设置同步
- **非目标**：不处理音频文件资源管理（当前使用系统音效），不处理音频编辑与合成

## 不变量与约束（改动护栏）

### 单位与坐标系

- 力度（power）使用 `Float` 类型，范围 0.0-1.0（归一化）
- 冲量（impulse）使用 `Float` 类型，单位未明确（物理引擎传递的值）
- 音量（volume）使用 `Float` 类型，范围 0.0-1.0

### 数值稳定性保护

- **力度阈值**：`playCueHit(power:)` 使用 0.3/0.7 作为 soft/medium/hard 分界，必须保持稳定
- **冲量阈值**：`playBallCollision(impulse:)` 使用 0.5/2.0 作为 light/medium/hard 分界，必须保持稳定
- **音量范围**：`volume` setter 中应限制范围，当前未实现（建议添加 `min(max(value, 0), 1)`）

### 时序与状态约束

- AVAudioSession 必须在 `init()` 中配置，确保应用启动时音频会话就绪
- `syncWithUserSettings(_:)` 应在设置变更时立即调用，确保音效状态与用户设置同步
- 震动反馈生成器应在每次触发前创建新实例（当前实现），避免状态污染

## 状态机 / 事件模型

```
AudioManager 初始化
  ↓
setupAudioSession() → 配置 AVAudioSession
  ↓
preloadSounds() → 预加载音效（当前为空）
  ↓
外部调用播放方法
  ↓
检查 isSoundEnabled → false 则返回
  ↓
播放系统音效
  ↓
检查 isHapticEnabled → false 则返回
  ↓
触发震动反馈
```

## 错误处理与降级策略

- **AVAudioSession 配置失败**：记录错误日志但不中断流程（音效可能仍可播放）
- **系统音效播放失败**：AudioServicesPlaySystemSound 不会抛出异常，失败时静默忽略
- **自定义音效文件加载失败**：记录错误日志，回退到系统音效（`playCustomSound` 中已实现）

## 性能考量

- 单例模式确保全局唯一实例，避免重复初始化
- 系统音效使用 AudioServicesPlaySystemSound，性能开销小
- 震动反馈生成器每次触发时创建新实例，避免状态污染但可能增加开销（如频繁触发可考虑缓存）

## 设计决策记录（ADR）

### 使用单例模式而非依赖注入
- **背景**：音效管理是全局服务，需要在多个模块中访问
- **候选方案**：单例模式（当前）、依赖注入、环境值
- **结论**：单例模式最简洁，符合工具类设计模式，AudioManager 无状态依赖
- **后果**：难以进行单元测试（需通过 UIApplication 模拟），但音效播放本身更适合集成测试

### 当前使用系统音效而非自定义音频文件
- **背景**：需要快速实现音效功能，自定义音频文件需要资源管理
- **候选方案**：系统音效（当前）、自定义音频文件、第三方音频库
- **结论**：系统音效实现简单，无需资源管理，适合 MVP 阶段
- **后果**：音效质量与多样性受限，后续可迁移到自定义音频文件（已提供 `loadCustomSound` 接口）

### 震动反馈每次创建新生成器
- **背景**：UIImpactFeedbackGenerator 需要 prepare 才能保证低延迟，但当前实现未调用 prepare
- **候选方案**：每次创建新实例（当前）、缓存生成器并调用 prepare、使用全局生成器
- **结论**：每次创建新实例最简单，避免状态污染
- **后果**：频繁触发时可能增加开销，如性能问题可考虑缓存 + prepare 优化
