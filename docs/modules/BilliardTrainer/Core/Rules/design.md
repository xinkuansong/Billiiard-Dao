# Rules（规则系统）- 设计与约束

> 对应 README：[README.md](./README.md)
> 文档最后更新：2026-02-27

## 设计目标与非目标

- **目标**：
  - 实现中式八球完整规则判定（开球、花色确定、清台、打8号球、犯规处理）
  - 支持球名格式兼容（`ball_N` 和 `_N`）
  - 提供清晰的状态机和状态消息
- **非目标**：
  - 不处理物理模拟（仅消费事件）
  - 不处理 UI 显示（仅提供状态数据）
  - 不实现其他台球规则（如9球、斯诺克）

## 不变量与约束（改动护栏）

### 单位与坐标系

- 事件时间单位为秒（s），但规则判定不依赖时间值，仅依赖事件类型和顺序
- 球号范围：1-7（全色球）、8（黑球）、9-15（花色球）

### 数值稳定性保护

- **球名解析**：`extractBallNumber()` 支持 `ball_N` 和 `_N` 两种格式，解析失败返回 `nil`，调用方需要处理
- **球号范围检查**：在判断球组时，检查球号是否在有效范围内（1-7 或 9-15），避免无效球号导致错误判定
- **空事件列表**：`isLegalShot()` 中，如果事件列表为空，返回 `(false, [.noBallHit])`，避免空指针异常

### 时序与状态约束

- **事件顺序**：`GameEvent` 数组应按时间顺序排列，`analyzeShotEvents()` 中按顺序处理，首碰球取第一个球球碰撞事件
- **状态转换顺序**：
  - `waitingBreak` → `openTable` / `playing(group)`（开球后）
  - `openTable` → `playing(group)`（确定花色后）
  - `playing(group)` → `eightBallStage`（清台后）
  - `eightBallStage` → `gameOver(won)`（打进8号球或犯规）
  - 任意阶段 → `gameOver(won)`（犯规导致游戏结束）
- **花色确定规则**：开球或花色未定时，如果进袋球只有全色球或只有花色球，则确定花色；如果都有或都没有，保持 `open` 状态
- **8号球规则**：
  - 花色未定时打进8号球 → 输
  - 己方花色未清台时打进8号球 → 输
  - 己方花色清台后打进8号球 → 赢
  - 犯规时打进8号球 → 输
- **自由球规则**：
  - 开球犯规 → `ballInHandBehindLine = true`（限制在开球线后）
  - 其他犯规 → `ballInHandBehindLine = false`（可自由放置）

### 常量与阈值清单

| 常量/阈值 | 值 | 来源/理由 | 改动影响 |
|-----------|-----|----------|----------|
| 全色球范围 | 1-7 | 中式八球规则 | 影响花色判定 |
| 花色球范围 | 9-15 | 中式八球规则 | 影响花色判定 |
| 8号球 | 8 | 中式八球规则 | 影响8号球判定 |
| 球名格式 | `ball_N` 或 `_N` | 兼容性要求 | 影响球名解析 |
| 白球名称 | `cueBall` | 约定 | 影响首碰球判定 |

## 状态机 / 事件模型

```
waitingBreak --[开球合法，进袋确定花色]--> playing(group)
waitingBreak --[开球合法，未确定花色]--> openTable
waitingBreak --[开球犯规]--> openTable（ballInHand）
openTable --[确定花色]--> playing(group)
openTable --[打进8号球]--> gameOver(won: false)
playing(group) --[清台]--> eightBallStage
playing(group) --[犯规时打进8号球]--> gameOver(won: false)
eightBallStage --[合法打进8号球]--> gameOver(won: true)
eightBallStage --[犯规]--> gameOver(won: false)
任意阶段 --[严重犯规]--> gameOver(won: false)
```

### 状态说明

- **waitingBreak（等待开球）**：初始状态，等待玩家开球
- **openTable（花色未定）**：开球后未确定花色，或开球犯规后
- **playing(group)（进行中）**：已确定花色，玩家击打己方球组
- **eightBallStage（打8号球）**：己方花色清台，必须打8号球
- **gameOver(won)（游戏结束）**：游戏结束，won 表示当前玩家是否获胜

## 错误处理与降级策略

- **球名解析失败**：`extractBallNumber()` 返回 `nil`，在判定花色时视为无效球，可能导致错误判定（但不会崩溃）
- **事件列表为空**：`isLegalShot()` 返回 `(false, [.noBallHit])`，`processShot()` 返回 `false`（不继续击球）
- **无效球号**：球号不在 1-15 范围内时，不参与花色判定，避免错误分组
- **状态转换异常**：如果状态转换不符合规则，`processShot()` 可能返回错误结果，需要调用方处理

## 性能考量

- **事件分析**：`analyzeShotEvents()` 遍历事件列表一次，复杂度 O(n)，n 为事件数量（通常 < 50），性能优秀
- **球组判定**：`tryAssignGroup()` 遍历进袋球列表，复杂度 O(m)，m 为进袋球数量（通常 < 8），性能优秀
- **状态更新**：`processShot()` 复杂度 O(1)，性能优秀

## 参考实现对照（如适用）

本模块不涉及物理引擎计算，无需对照 pooltool 参考实现。但规则判定需要遵循中式八球标准规则。

## 设计决策记录（ADR）

### ADR-001：使用状态机管理游戏阶段

- **背景**：游戏有多个阶段（开球、花色未定、进行中、打8号球、结束），需要清晰的状态管理
- **候选方案**：
  1. 使用状态机（当前方案）
  2. 使用标志位组合
  3. 使用命令模式
- **结论**：选择状态机方案，因为阶段转换有明确的规则约束，状态机能清晰表达这些约束
- **后果**：状态转换必须通过 `processShot()`，不能直接修改状态

### ADR-002：球名格式兼容

- **背景**：系统中存在两种球名格式（`ball_N` 和 `_N`），需要兼容
- **候选方案**：
  1. 统一转换为一种格式（需要修改所有调用方）
  2. 在规则模块中兼容两种格式（当前方案）
  3. 不兼容，要求统一格式
- **结论**：选择在规则模块中兼容两种格式，因为这是最小改动方案，不影响其他模块
- **后果**：`extractBallNumber()` 需要处理两种格式，代码稍复杂

### ADR-003：事件驱动设计

- **背景**：规则判定需要基于物理事件（碰撞、进袋等）
- **候选方案**：
  1. 事件驱动（当前方案）：消费 `GameEvent` 数组
  2. 轮询物理状态
  3. 回调通知
- **结论**：选择事件驱动方案，因为事件已经包含所有需要的信息，且时序清晰
- **后果**：需要确保事件顺序正确，事件格式稳定

### ADR-004：分离规则判定和状态管理

- **背景**：规则判定（`isLegalShot()`）和状态管理（`processShot()`）可以分离
- **候选方案**：
  1. 分离（当前方案）：`GameRules` 提供规则判定，`EightBallGameManager` 管理状态
  2. 合并到一个类
- **结论**：选择分离方案，因为规则判定可以独立测试和复用，状态管理可以独立演进
- **后果**：需要确保两个模块的接口一致，避免重复逻辑
